OutfitKit

Guía de uso

@outfitkit/core es una librería de Web Components (Lit) que envuelve Ionic por completo: tu código usa solo ok-* y nunca ion-* directamente. Se distribuye standalone por npm y CDN, con imports individuales o bundle.

Requisito común: OutfitKit asume que el host registra @ionic/core una sola vez (los ok-* esconden ion-* por dentro) y que lit está disponible (bundler o import-map). OutfitKit deja lit como external para que haya una sola copia.
Instalación con npm Para apps con bundler (Vite, webpack, Ionic React, etc.). 
npm i @outfitkit/core @ionic/core lit

Import individual por componente (recomendado, tree-shake)

// registra solo lo que usas
import '@outfitkit/core/ok-button';
import '@outfitkit/core/ok-data-table';
import '@outfitkit/core/ok-app-shell';

// los tipos también se exportan desde el barrel
import type { OkNavSection, OkHeaderAction } from '@outfitkit/core';

Bundle completo (registra todos los ok-*)

// un solo import auto-registra TODA la librería
import '@outfitkit/core/cdn';

El tema (tokens) por separado

// plantilla de tokens --ok-* (cópiala/ajústala a tu marca)
import '@outfitkit/core/theme.css';
 Uso por CDN (sin build) Para landings, prototipos o plantillas Django sueltas. Es exactamente lo que hace este showcase. 
<!-- Ionic Core (registra los ion-* internos) + tema -->
<script type="module" src="https://cdn.jsdelivr.net/npm/@ionic/core/dist/ionic/ionic.esm.js"></script>
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/@ionic/core/css/ionic.bundle.css" />
<script type="module" src="https://cdn.jsdelivr.net/npm/ionicons/dist/ionicons/ionicons.esm.js"></script>

<!-- lit como external vía import-map -->
<script type="importmap">
  { "imports": { "lit": "https://esm.run/lit@3", "lit/": "https://esm.run/lit@3/" } }
</script>

<!-- OutfitKit: tokens + bundle (servido p.ej. desde unpkg/jsdelivr o tu propio CDN) -->
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/@outfitkit/core/dist/theme.example.css" />
<script type="module" src="https://cdn.jsdelivr.net/npm/@outfitkit/core/dist/outfitkit.js"></script>
CSP estricta: la salida no contiene eval/new Function, así que funciona bajo script-src 'self' (Lit usa tagged templates + adoptedStyleSheets). Para un CDN propio bajo CSP, sírvelo desde tu mismo origen o añade el host del CDN a la directiva.
Tematizar con tokens --ok-* Theming en dos capas, estilo Ionic.

Capa 1: tokens globales --ok-* en tu :root (espejo de --ion-*). Si hay Ionic cargado, lo más cómodo es heredarlos para que el tema claro/oscuro se aplique solo: 

:root {
  --ok-primary: var(--ion-color-primary);
  --ok-bg: var(--ion-background-color);
  --ok-surface: var(--ion-card-background);
  --ok-text: var(--ion-text-color);
  --ok-border: rgba(var(--ion-text-color-rgb), 0.12);
  --ok-radius: 12px;
  --ok-spacing: 1rem;
  --ok-container-max: 1140px;
  --ok-font: system-ui, sans-serif;
}

Sin Ionic (p.ej. una landing Django sin app Ionic), pon valores propios de marca:

:root {
  --ok-primary: #1496d6;
  --ok-bg: #ffffff;
  --ok-surface: #ffffff;
  --ok-text: #18181b;
  --ok-border: rgba(24, 24, 27, 0.12);
}

Capa 2: vars por componente, estilo Ionic y sin guion bajo (--background, --color, --border-radius, --padding…). Cada una cae en cadena --ok-* → --ion-* → hex. Así puedes ajustar un componente concreto igual que ion-button: 

ok-button { --background: #7c5cff; --border-radius: 999px; }
ok-input  { --border-radius: 6px; }

 Uso en Django Carga las dependencias con <script type="module"> y, bajo CSP estricta, añade el nonce a tus scripts inline.

El bundle de OutfitKit es un fichero externo (no necesita nonce). El property binding (p.ej. .rows de ok-data-table) va en un script con nonce: 

<link rel="stylesheet" href="{% static 'vendor/outfitkit/theme.css' %}" />
<script type="module" src="{% static 'vendor/outfitkit/outfitkit.js' %}"></script>

<ok-data-table id="tabla"></ok-data-table>

<script type="module" nonce="{{ request.csp_nonce }}">
  const dt = document.getElementById('tabla');
  dt.columns = {{ columns_json|safe }};
  dt.rows = {{ rows_json|safe }};
  dt.addEventListener('rowAction', (e) => {
    // POST al servidor con el id de la fila
  });
</script>
Sirve los assets desde tu propio origen (static/) para cumplir script-src 'self'. El import-map de lit también debe apuntar a un origen permitido por tu CSP (o vendoriza lit en static/).
Uso en Lit / Hub Importa los ok-* que uses y cablea props/eventos con la sintaxis declarativa de Lit (.prop y @evento). 
import { LitElement, html } from 'lit';
import '@outfitkit/core/ok-data-table';
import '@outfitkit/core/ok-button';

class Productos extends LitElement {
  rows = [{ id: 1, name: 'Café', price: 1.2 }];
  render() {
    return html`
      <ok-data-table
        .columns=${[{ key: 'name', header: 'Producto' }]}
        .rows=${this.rows}
        @rowAction=${(e) => this.onRow(e.detail)}></ok-data-table>
    `;
  }
}
Property binding con .columns/.rows (no atributos): son datos JS, no strings. Los eventos llegan ya normalizados como ok-* (p.ej. @ok-change, @rowAction); nunca escuches ionChange.

 Reglas de oro
  • Usa solo ok-*; nunca ion-* directamente.
  • Datos por property binding (.columns, .rows, sidebar.sections, topbar.actions, segment.items, select.options), no atributos.
  • Escucha siempre eventos ok-*; los de Ionic se re-emiten con nombre propio.
  • Navegación sin router: ok-sidebar emite ok-nav { path }; tú actualizas active-path y enrutas (este showcase navega con location.href).
